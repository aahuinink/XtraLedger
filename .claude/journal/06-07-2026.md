## 2026-07-05 (Sun)

**Work done:**
- Question: is the type usage (`uint16_t` slots/capacity, `uint32_t` hash, `int32_t` +
  `-1` sentinel on the public lookup API) typical/professional for this use case? Discussed:
  fixed-width types are the right call given the deliberate `UINT16_MAX` capacity cap;
  widening to a signed type one size up to carry a `-1` sentinel is a named, standard idiom
  (POSIX `ssize_t`, `ungetc`'s `int` return), not a hack. Flagged the two-layer "not found"
  encoding (`get()`'s internal `UINT32_MAX - index` translated to the public `-1`) as
  something to keep deliberate rather than accidental, and noted `uint_fast16_t` mixed into
  otherwise fixed-width code in `get()`'s loop counter as worth a conscious yes/no.

**Bugs found (by review):**
1. **Real linkage bug**: `get()` and `put()` had external linkage (no `static`) while every
   other private helper in the file (`hash`, `hash_to_index`, `determine_required_capacity`,
   `increase_capacity_if_needed`) was already `static`. `get`/`put` are about as generic as C
   symbol names get — a future translation unit defining its own would either fail to link or,
   worse, silently collide. Fixed same session (`3e6bd10`).
2. Minor: a comment gave the empty-string DJB2 hash as "5361" instead of the correct 5381.
   Fixed same commit.
3. Minor: `arena_lookup_try_update`'s `uint16_t` return was captured into an `int32_t` local
   in one test — harmless widening, but sloppy typing. Fixed same commit.

**Mistakes / things to watch:**
