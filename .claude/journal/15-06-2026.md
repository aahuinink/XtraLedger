## Critical bugs (these are why I want you to re-open the resize path)

### 1. Resize never rehashes â€” the table is silently corrupt after the first grow
`populate_lookup_table()` only inserts the `num_new_keys` newest keys, starting at
`table->size`. But every existing entry's slot was computed as `hash & (old_capacity-1)`.
When capacity changes (8â†’16), `hash_to_index` returns *different* indices, so every
previously-inserted key is now in the wrong bucket. Open addressing requires you to
**reinsert every entry** on resize.

Your test passes only because it never re-looks-up the first 5 keys *after* the grow to 16
(`lookup-table.c:55-58` checks `capacity`/`size`, not retrieval). Add this and watch it fail:
```c
for (uint_fast8_t i = 0; i < 8; ++i)
    assert(arena_lookup_try_get(&table, &test_keys[i]) == i);   // will break for old keys
```
This is the single most important lesson of the week: **a green test suite is only as
strong as its assertions.** A passing test that doesn't assert the property you care about
is worse than no test, because it buys false confidence.

### 2. The resize `memcpy` is broken three independent ways (`arena-lookup-table.c:225`)
```c
memcpy(table->entries, new_entry_array, table->capacity);
```
- **src/dest swapped:** this copies the *uninitialized new* array *into the old* array,
  then you set `entries = new_entry_array` (still uninitialized) and free the old one. The
  old data is thrown away; the new array is never populated.
- **wrong length:** `table->capacity` is an element count (~16), used here as a *byte*
  count, and it's the *new* capacity, not the old occupied size. You'd want
  `old_count * sizeof(struct arena_lookup_entry)` at most.
- **wrong operation:** even a correct copy is invalid for a hash table â€” see bug #1, you
  must rehash, not blit. Once you rehash, this memcpy disappears entirely.

### 3. Reading uninitialized heap memory is UB, not just "unlikely" (`:164-165`, `:210`)
`malloc` returns indeterminate bytes; you then read `.state` and compare against
`ENTRY_STATE_ACTIVE`. Your comment frames the `0xF364` sentinel as "highly unlikely to
collide" â€” but the real problem isn't collision odds, it's that *reading indeterminate
memory is undefined behavior*, full stop. Compilers are allowed to assume it never happens.
Fix: `calloc` (or `memset` after `malloc`) and make `state == 0` mean "empty". Then the
sentinel-collision worry evaporates because you control the bytes. Cheaper and correct.

### 4. `arena_lookup_try_get` can infinite-loop on a full table (`:82`)
`while (true)` only exits when it finds a non-ACTIVE slot or a hash match. If the table is
ever 100% ACTIVE (no empty sentinel) and the key is absent, it spins forever. The 0.7 load
factor *should* guarantee an empty slot â€” but only if resize (bug #1/#2) actually works.
Defense in depth: bound the probe count by `capacity`.

## Design feedback (account.h)

### `const` struct members make your out-params unwritable
```c
struct xl_account_snapshot {
    int32_t balance;                    // not const
    const enum xl_acct_normality norm;  // const
    const xl_smallstr64 name;           // const
    const xl_smallstr128 desc;          // const
};
```
A snapshot is filled via `..._snapshot(..., struct xl_account_snapshot * out)`. To populate
it the implementation must assign to its members â€” but `const` members can only be set by an
initializer at object creation, not by later assignment (`out->name = ...` is ill-formed).
You'll either fight the compiler with casts (UB if the object is genuinely const) or be
unable to write the snapshot at all. **Const-ness of a *copy* should live with the
variable, not baked into the type.** Drop the member `const`s; let callers declare
`const struct xl_account_snapshot snap`.

### View vs. snapshot inconsistency
In `xl_account_view`, `balance`/`name`/`desc` are *pointers* into the arena (that's what
gives the "lifetime guarantee"), but `norm` is a plain value `const enum xl_acct_normality`.
A copied enum has no lifetime relationship to the arena â€” it defeats the view's purpose and
is inconsistent with its siblings. Make it `const enum xl_acct_normality * norm`.

### Smaller notes
- `try_get_account_name_by_handle(..., const xl_smallstr64 * out_str)` â€” an *out* parameter
  that's `const` can't be written to. Same category as the snapshot issue.
- Doc cleanup (`@return`, ERRORS sections) is a real improvement â€” keep doing this. A few
  ERRORS blocks are still empty stubs; fill or remove them so they don't rot.

## Style / project-fit

Your CLAUDE.md lists "prioritize readability over cleverness." Several spots trade
readability for branchless micro-tricks that the compiler would do anyway:
- `c += 32 * ((c <= 'Z') & (c >= 'A'));` (hash lowercasing)
- `index *= (index < table->capacity);` (probe wraparound)
- `hash & (table_capacity - (table_capacity != UINT16_MAX))` (mask)

None of these are on a measured hot path yet. Write the obvious `if`/`% capacity` version,
*then* reach for tricks only when a profile says this function matters. You're explicitly
here to learn profiling â€” so let measurement, not intuition, justify cleverness. (The
`UINT16_MAX` edge case in the mask is genuinely subtle; a one-line comment on *why* it's
there would help future-you.)

Naming: files are kebab-case (`arena-lookup-table.c`) but `account_store.{c,h}` /
`account-arena` mix kebab and snake. Pick one convention for filenames.
