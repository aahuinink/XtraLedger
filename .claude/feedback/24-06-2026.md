## `arena-lookup-table.c` — correctness (must fix today)

1. **`increase_capacity_if_needed` is broken three ways — the resize loses all data.**
   Look hard at lines 207, 225. Trace it with a debugger or on paper:
   - `memcpy(table->entries, new_entry_array, table->capacity)` — check the **argument order**
     of `memcpy(dest, src, n)`. Which way is the data flowing? Which buffer gets freed two lines
     later? Which one does `table->entries` end up pointing at?
   - The **count** is wrong too. `table->capacity` is a *number of entries*, and memcpy wants
     *bytes*. What's missing? (compare to the `malloc` on line 210.)
   - Line 207 sets `table->capacity = new_min_capacity` **before** the copy. If you were copying
     the old data, which capacity should bound the copy — old or new?

2. **The rehash never happens — this is the Q1 bug.** Even after you fix the copy above, the
   old entries are still sitting at indices computed with the *old* mask. `populate_lookup_table`
   only walks `num_new_keys` starting at `table->size`; it never touches existing entries. Your
   Q1 answer has the right idea ("iterate every non-empty entry, recompute index") — now make the
   code match it. Decide: on a grow, do you (a) allocate zeroed, then reinsert *every* live entry
   into the fresh array, or (b) try to shuffle in place? (a) is simpler and harder to get wrong.
   **Constraint from Q1: use the stored `entry.hash`, do not re-run DJB2 on the reinsert path.**
   That means your reinsert helper takes an `arena_lookup_entry`, not a key string.

3. **`malloc` + reading `.state` is UB — the Q3 bug.** Lines 210 + 164–165. The `0xF364`
   sentinel only "works" if uninitialized memory happens not to contain it — but reading that
   memory at all is undefined. Switch to `calloc`, let `state == 0` mean empty, delete
   `ENTRY_STATE_ACTIVE = 0xF364`. Your Q3 answer already argues this; land it.

4. **`arena_lookup_try_get` can loop forever (review bug #4).** `while (true)` on line 82 only
   exits on an empty slot or a match. If the table were ever full (or a bug fills it), this spins.
   Bound the probe by capacity: after at most `capacity` probes, give up and return not-found.

5. **Dead prototype.** `get_starting_entry` (line 29) is declared, never defined or used. Remove it.

## `arena-lookup-table.c` — minor / note for later
- `hash()` uses `char c`; on signed-char platforms a byte ≥ 0x80 makes the `(c >= 'A')` test
  and the DJB2 accumulation behave oddly. Account names are ASCII today, so park it — but a
  one-line `unsigned char` would remove the footgun.
- `try_get` compares `XL_SMALLSTR64_PAYLOAD_SIZE` bytes with `strncasecmp`. Two keys sharing a
  prefix but differing in length could miscompare against trailing garbage. Compare lengths first,
  or null-terminate the stored side. Note it; not today's fight.

## `account.h` — the Q4 fix hasn't landed
Your Q4 answer correctly says `const` members make the snapshot out-param unwritable — but
`account.h:29–31` **still has `const` on the snapshot members**, so `try_get_*_snapshot` still
can't be implemented. Remove `const` from the *struct fields*; constness belongs on the
*caller's pointer*, exactly as your Q4 code sketch shows. Separately, `xl_account_view:37` has
`norm` by value while its siblings are pointers — a view is supposed to be all borrowed
pointers. Make it consistent (`const enum xl_acct_normality * norm`). Defer the full header
review; just unblock the struct so the accessors are implementable.
