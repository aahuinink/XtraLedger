## Blocking bug: `put()` treats every insert as a duplicate

`entry_index` is declared `uint16_t` but assigned from `get()`, which returns `int32_t` and
encodes "not found, insert here" as a **negative** index:

```c
uint16_t entry_index = get(table, &table->keys[slot], key_hash);
if (entry_index >= 0) {
    xl_errno = XL_EDUPL;
    return false;
}
```

Assigning a negative `int32_t` into a `uint16_t` wraps it into a huge unsigned value. `entry_index
>= 0` is then *always* true for an unsigned type — every single insert takes the "duplicate found"
branch and fails, including the first one on an empty table. That's the whole table load failing,
which is what you're seeing.

Fix the type so the sign survives the assignment.

## Design flaw underneath that: the negative-index encoding is ambiguous at index 0

Even once the type is fixed, look at `get()`:

```c
if ( possible_match.hash == 0x0 ) {
    return -1 * possible_match_index;
}
...
return slot;   // a real match, also just a plain non-negative int
```

If the empty slot found is at index 0, `-1 * 0 == 0` — indistinguishable from "matched at slot 0."
`put()` can't tell "insert at index 0" from "found a real match at slot 0" and will silently
corrupt or misbehave depending on which branch runs. This won't show up in your current test
(unlucky to land exactly on index 0) but it's latent and will bite you as an intermittent,
hash-dependent failure. Think about how to make "not found" unambiguous from "found" — the sign
trick doesn't work at zero.

## Wrong error code: `arena_lookup_try_get`

```c
if ( slot < 0 ) {
    xl_errno = XL_EDUPL;
    return -1;
}
```

`XL_EDUPL` means "a duplicate exists" — that's `put()`'s error. Here you're looking up a key that
isn't present; per your own convention (see `account.h`'s use of `XL_ENFND`), this should be
`XL_ENFND`.
